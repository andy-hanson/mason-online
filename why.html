mason

why?setuptour

Mason is designed to reduce the syntax of JavaScript (JS) and expand its capabilities.

Its goals are:

  • Be syntactically light. No { } ( ) , ; all over the place.

  • Everything JS can do, Mason can do better.

    This means the core language is just JavaScript minus syntax. Mason functions are JS functions, Mason classes are JS classes, Mason modules are JS modules.

    You can convert a JS project to Mason without changing the design at all.

  • Mason uses a mandatory standard library a small one called msl to provide basic functionality missing from JS. Many of msl's functions are builtins that are implicitly imported. For example, JS's == can't be overloaded, so Mason programs use the =? method from msl instead.
  • Don't impose any style. Mason isn't X-oriented for any X. This leads to it being bigger and less "pure", but more useful as a JS substitute. It adds new features but doesn't force you to use them.

So what features does it add? You'll read them all in the tour, but here's a preview:

  • Build syntax for constructing objects, lists, and maps. It's also a nice syntax for declarative data.
  • Most statements are also values. For example, if returns an optional value.
  • Assertions
  • Method functions: functions that can be polymorphically implemented by many types, including builtin types.
  • Kinds: abstract types that support multiple inheritance and can be safely implemented by builtin types.
  • Type declaration syntax, with dynamic type checks.
  • Async functions

Planned features:

  • HTML construction syntax (like JSX but not bound to any framework, good for both client and server)
  • Discriminated union syntax (actually just a class heirarchy)
  • Class properties
  • Decorators
  • Lazy builders (useful for lazy @for loops)

It's under active development here.
